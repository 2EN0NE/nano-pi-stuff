# 测试驱动开发

TDD 是红 → 绿的循环。本技能是让该循环产出值得保留的测试的参考：什么是好测试、测试放哪里、反模式、以及循环的规则。每个环节在每个周期都适用——在周期之前和期间查阅，而不是之后。

在探索代码库时，阅读 `CONTEXT.md`（如果存在），使测试名称和接口词汇与项目的领域语言一致，并尊重所涉及区域中的 ADR。

## 什么是好测试

测试通过公开接口验证行为，而非实现细节。代码可以完全改变；测试不应随之改变。好测试读起来像一份规范——"用户能用有效购物车结账"精确告诉你存在什么能力——并且能在重构中存活，因为它不关心内部结构。

参见 [tests.md](tests.md) 获取示例，以及 [mocking.md](mocking.md) 获取 mock 指南。

## Seam——测试放哪里

**Seam**（接缝）是你进行测试的公开边界：你在不窥探内部的情况下观察行为的接口。测试活在 seam 处，绝不对内部进行测试。

**只在预先约定的 seam 处测试。** 在写任何测试之前，写下待测的 seam 并与用户确认。不要在任何未确认的 seam 处编写测试。你无法测试所有东西——提前约定 seam 是让测试精力落在关键路径和复杂逻辑上，而非每个边缘情况。

提问："公开接口是什么？我们应该在哪些 seam 测试？"

## 反模式

- **实现耦合**——mock 内部协作者、测试私有方法、或通过侧信道验证（查询数据库而不是使用接口）。识别特征：重构时测试中断但行为未变。
- **同义反复**——断言用与代码相同的方式重新计算期望值（`expect(add(a, b)).toBe(a + b)`，或用相同方式手动推导的快照，或断言与自身相等的常量），因此它在构造上必然通过，永远无法与代码不一致。期望值必须来自独立的真相来源——已知良好的字面量、手动推演的例子、规范。
- **水平切分**——先写完所有测试，再写所有实现。批量测试验证的是_想象中_的行为：你测试的是事物的_形状_而非面向用户的行为，测试对真实变化不敏感，并且在理解实现之前就提交了测试结构。应当以**垂直切片**的方式工作——一个测试 → 一个实现 → 重复，每个测试都是一颗**示踪弹**，对上一周期教会你的东西做出响应。

## 循环的规则

- **先红后绿。** 先写失败的测试，然后只写足以通过它的代码。不要预判未来的测试或添加推测性功能。
- **一次一个切片。** 每个周期一个 seam、一个测试、一个最小实现。
- **重构不是循环的一部分。** 重构属于审查阶段（见 `code-review` 技能），而非红 → 绿实现周期。
