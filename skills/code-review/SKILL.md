name: code-review
description: 审查自某个固定点（commit、分支、标签或 merge-base）以来的变更，沿两个轴进行——规范（代码是否符合本仓库记录的编码标准？）和 spec（代码是否忠实实现了原始 issue/PRD 的要求？）。两个审查在并行子代理中运行并排报告。当用户想要审查分支、PR、正在进行的变更，或要求"审查自 X 以来的变更"时使用。
---

对 `HEAD` 与用户提供的固定点之间的 diff 进行双轴审查：

- **规范**（Standards）——代码是否符合本仓库记录的编码标准？
- **Spec**——代码是否忠实实现了原始 issue / PRD / spec？

两个轴作为**并行子代理**运行，以免污染彼此的上下文，然后本技能汇总它们的发现。

issue tracker 应已提供给你——如果 `docs/agents/issue-tracker.md` 缺失，运行 `/setup-matt-pocock-skills`。

## 流程

### 1. 确定固定点

用户说的就是固定点——一个 commit SHA、分支名、标签、`main`、`HEAD~5` 等。如果用户没有指定，请询问。

记录 diff 命令：`git diff <fixed-point>...HEAD`（三点，比较针对 merge-base）。同时通过 `git log <fixed-point>..HEAD --oneline` 记录提交列表。

在继续之前，确认固定点可解析（`git rev-parse <fixed-point>`）且 diff 非空。坏的引用或空 diff 应在此处失败——而不是在两个并行子代理内部。

### 2. 识别 spec 来源

按以下顺序查找原始 spec：

1. 提交消息中的 issue 引用（`#123`、`Closes #45`、GitLab `!67` 等）——通过 `docs/agents/issue-tracker.md` 中的工作流获取。
2. 用户作为参数传递的路径。
3. `docs/`、`specs/` 或 `.scratch/` 下与分支名或功能匹配的 PRD/spec 文件。
4. 如果找不到，询问用户 spec 在哪里。如果他们说没有，**Spec** 子代理将跳过并报告"无可用的 spec"。

### 3. 识别规范来源

仓库中记录编码方式的所有内容，如 `CODING_STANDARDS.md` 或 `CONTRIBUTING.md`。

在仓库所记录的任何内容之上，规范轴始终携带下面的**代码坏味基线**（smell baseline）——一组固定的 Fowler 代码坏味（《重构》第 3 章），即使仓库没有任何文档也适用。两条规则约束它：

- **仓库覆盖。** 仓库记录的标准始终优先；如果它认可了基线会标记的内容，则压制该坏味。
- **始终是判断。** 每个坏味都是一个标记了标签的启发式方法（"可能的依恋情结"），绝不是硬性违规——并且，像这里的任何标准一样，跳过任何工具已强制执行的内容。

每个坏味的格式为*它是什么* → _如何修复_；对照 diff 进行检查：

- **神秘命名（Mysterious Name）**——函数、变量或类型的名称不揭示其作用或内容。→ 重命名；如果找不到诚实的名称，设计还不清晰。
- **重复代码（Duplicated Code）**——相同的逻辑形状出现在变更中的多个块或文件中。→ 提取共享形状，从两处调用。
- **依恋情结（Feature Envy）**——方法访问其他对象的数据多于自身的。→ 将方法移到它所依恋的数据上。
- **数据泥团（Data Clumps）**——相同的几个字段或参数总是一起出现（一个待诞生的类型）。→ 将它们打包到一个类型中，传递那个类型。
- **基本类型偏执（Primitive Obsession）**——基本类型或字符串代表一个值得拥有自己类型的领域概念。→ 给这个概念自己的小型类型。
- **重复的 switch（Repeated Switches）**——同一类型上的 switch/if-级联在变更中反复出现。→ 用多态或一个两者共享的映射替换。
- **散弹式修改（Shotgun Surgery）**——一个逻辑变更迫使在许多文件中进行分散的编辑。→ 将一起变更的内容聚集到一个模块中。
- **发散式变更（Divergent Change）**——一个文件或模块因多个不相关的原因而被编辑。→ 拆分，使每个模块因一个原因而变更。
- **推测性通用化（Speculative Generality）**——为了 spec 中不存在的需求而添加的抽象、参数或钩子。→ 删除它；内联回去，直到真正的需求出现。
- **消息链（Message Chains）**——调用者不应依赖的长 `a.b().c().d()` 导航。→ 在第一个对象上用一个方法隐藏遍历过程。
- **中间人（Middle Man）**——一个主要委托给其他对象的类或函数。→ 砍掉它，直接调用真正的目标。
- **拒绝遗产（Refused Bequest）**——子类或实现者忽略或覆盖了它继承的大部分内容。→ 放弃继承，使用组合。

### 4. 并行生成两个子代理

发送一条消息，包含两个 `Agent` 工具调用。对两者使用 `general-purpose` 子代理。

**规范子代理提示**——包含：

- 完整的 diff 命令和提交列表。
- 你在步骤 3 中找到的规范源文件列表，**以及步骤 3 中的坏味基线全部粘贴进去**——子代理没有其他方式访问它。
- 简报："报告——按相关文件/hunk——(a) diff 违反记录标准的每一处：引用标准（文件 + 规则）；以及 (b) 你发现的任何基线坏味：指出名称并引用 hunk。区分硬性违规与判断——记录标准违规可能是硬性的，但基线坏味始终是判断，且记录的标准覆盖基线。跳过工具已强制执行的任何内容。400 词以内。"

**Spec 子代理提示**——包含：

- diff 命令和提交列表。
- spec 的路径或获取的内容。
- 简报："报告：(a) spec 要求但缺失或不完整的需求；(b) diff 中未被要求的行为（范围蔓延）；(c) 看起来已实现但实现方式有误的需求。每个发现引用 spec 的行。400 词以内。"

如果 spec 缺失，跳过 Spec 子代理并在最终报告中注明。

### 5. 汇总

在两个标题 `## 规范` 和 `## Spec` 下呈现两份报告，逐字或稍作清理。**不要**合并或重新排级发现——这两个轴是刻意分开的（见下文_为什么有两个轴_）。

以一行摘要结束：每个轴的总发现数，以及_每个轴内_最严重的问题（如果有）。不要跨轴选一个优胜者——那正是分离要防止的重新排级。

## 为什么有两个轴

一个变更可能通过一个轴而失败于另一个：

- 代码遵循每个标准但实现了错误的东西 → **规范通过，Spec 失败。**
- 代码精确地做了 issue 要求的但破坏了项目约定 → **Spec 通过，规范失败。**

分别报告它们防止一个轴掩盖另一个。
