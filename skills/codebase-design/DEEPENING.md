# 深化

如何安全地深化一个浅模块集群，给定其依赖关系。假定 [SKILL.md](SKILL.md) 中的词汇——**module**、**interface**、**seam**、**adapter**。

## 依赖类别

在评估深化候选时，对其依赖进行分类。类别决定了深化后的模块如何在其 seam 处进行测试。

### 1. 进程内（In-process）

纯计算、内存状态、无 I/O。始终可深化——合并模块并通过新接口直接测试。无需适配器。

### 2. 本地可替代（Local-substitutable）

有本地测试替代品的依赖（PGLite 替代 Postgres、内存文件系统）。如果替代品存在则可深化。深化后的模块在测试套件中使用替代品运行进行测试。seam 是内部的；模块外部接口处没有 port。

### 3. 远程但自有（Ports & Adapters）

你自己的跨网络边界的服务（微服务、内部 API）。在 seam 处定义一个 **port**（接口）。深模块拥有逻辑；传输作为**适配器**注入。测试使用内存适配器。生产环境使用 HTTP/gRPC/队列适配器。

推荐写法：_"在 seam 处定义一个 port，为生产环境实现 HTTP 适配器，为测试实现内存适配器，使逻辑位于一个深模块中，即使它跨网络部署。"_

### 4. 真正的外部（Mock）

你无法控制的第三方服务（Stripe、Twilio 等）。深化后的模块将外部依赖作为注入的 port 接收；测试提供 mock 适配器。

## Seam 纪律

- **一个适配器 = 假设性的 seam。两个适配器 = 真实的 seam。** 除非至少有两个适配器是合理的（通常是生产 + 测试），否则不要引入 port。单一适配器的 seam 仅仅是间接层。
- **内部 seam vs 外部 seam。** 一个深模块可以有内部 seam（对其实现私有，用于自身测试）以及其接口处的外部 seam。不要仅仅因为测试使用就将内部 seam 暴露到接口中。

## 测试策略：替换，而非分层

- 一旦深化模块接口处存在测试，旧的浅模块单元测试就变成浪费——删除它们。
- 在深化模块的接口处编写新测试。**接口就是测试表面**。
- 测试断言通过接口可观察的结果，而非内部状态。
- 测试应能经受内部重构——它们描述行为，而非实现。如果测试在实现变化时必须改变，说明它绕过了接口在测试。
