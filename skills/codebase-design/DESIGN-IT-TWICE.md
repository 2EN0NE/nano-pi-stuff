# 设计两次

当用户想为选定的深化候选项探索替代接口时，使用此并行子代理模式。基于"Design It Twice"（Ousterhout）——你的第一个想法不太可能是最好的。

使用 [SKILL.md](SKILL.md) 中的词汇——**module**、**interface**、**seam**、**adapter**、**leverage**。

## 流程

### 1. 框定问题空间

在生成子代理之前，为所选候选项编写面向用户的问题空间说明：

- 任何新接口需要满足的约束
- 它将依赖什么，以及它们属于哪个类别（见 [DEEPENING.md](DEEPENING.md)）
- 一个粗略的说明性代码草图，使约束具体化——不是提案，只是使约束具象化的方式

向用户展示，然后立即进入步骤 2。用户在子代理并行工作时阅读和思考。

### 2. 生成子代理

使用 Agent 工具并行生成 3+ 个子代理。每个必须为深化模块生成一个**根本不同的**接口。

用独立的技术简报提示每个子代理（文件路径、耦合细节、[DEEPENING.md](DEEPENING.md) 中的依赖类别、seam 后面是什么）。简报与步骤 1 中面向用户的问题空间说明无关。给每个代理不同的设计约束：

- Agent 1："最小化接口——最多 1-3 个入口点。最大化每个入口点的杠杆效应。"
- Agent 2："最大化灵活性——支持更多用例和扩展。"
- Agent 3："为最常见的调用者优化——使默认情况变得非常简单。"
- Agent 4（如适用）："围绕 ports & adapters 设计跨 seam 依赖。"

在简报中同时包含 [SKILL.md](SKILL.md) 词汇和 CONTEXT.md 词汇，使每个子代理的命名与架构语言和项目领域语言一致。

每个子代理输出：

1. 接口（类型、方法、参数——以及不变量、排序、错误模式）
2. 使用示例，展示调用者如何使用
3. 实现隐藏在 seam 后面的是什么
4. 依赖策略和适配器（见 [DEEPENING.md](DEEPENING.md)）
5. 权衡——哪里杠杆效应高，哪里薄弱

### 3. 呈现和比较

依次呈现设计，使用户能吸收每个，然后用散文对比。通过**深度**（接口处的杠杆效应）、**局部性**（变更集中的位置）和 **seam 放置**进行对比。

比较后，给出你自己的推荐：哪个设计最强以及为什么。如果不同设计中的元素可以很好地组合，提出混合方案。要有主见——用户想要的是强判断，而不是菜单。
