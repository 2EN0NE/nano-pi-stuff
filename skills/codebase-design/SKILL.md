# 代码库设计

设计**深模块**：在小接口背后隐藏大量行为，放置在干净的 seam 处，通过该接口可测试。在任何设计或重构代码的地方使用这些语言和原则。目标是调用者的杠杆效应、维护者的局部性、以及每个人的可测试性。

## 词汇表

精确使用这些术语——不要用 "component"、"service"、"API" 或 "boundary" 替代。一致的语言是全部意义所在。

**模块（Module）** ——任何具有接口和实现的东西。有意的规模无关性：一个函数、类、包或跨层切片。_避免_：unit、component、service。

**接口（Interface）** ——调用者必须知道的正确使用模块的所有内容：类型签名，还有不变量、排序约束、错误模式、所需配置和性能特征。_避免_：API、signature（过于狭窄——它们只指向类型层面）。

**实现（Implementation）** ——模块内部的内容。与**适配器（Adapter）**不同：某个东西可以是一个小型适配器带大量实现（Postgres repo），或一个大型适配器带少量实现（内存 fake）。当 seam 是主题时用 "adapter"；否则用 "implementation"。

**深度（Depth）** ——接口处的杠杆效应：调用者（或测试）每学习一单位接口能行使的行为量。模块在小型接口后藏大量行为时为**深（deep）**，在接口几乎和实现一样复杂时为**浅（shallow）**。

**Seam（接缝）** _(Michael Feathers)_ ——可以在不编辑该位置的情况下改变行为的地方；模块接口所在的_位置_。seam 放在哪里本身就是设计决策，与放在它后面的是什么不同。_避免_：boundary（与 DDD 的 bounded context 过载）。

**适配器（Adapter）** ——在 seam 处满足接口的具体东西。描述_角色_（它填充什么插槽），而非实质（里面是什么）。

**杠杆效应（Leverage）** ——调用者从深度中获得的东西：每学习一单位接口获得更多能力。一个实现回馈 N 个调用点和 M 个测试。

**局部性（Locality）** ——维护者从深度中获得的东西：变更、bug、知识和验证集中在一个地方，而不是散布在调用者之间。修复一次，到处修复。

## 深 vs 浅

**深模块** = 小接口 + 大量实现：

```
┌─────────────────────┐
│   小型接口           │  ← 方法少、参数简单
├─────────────────────┤
│                     │
│  深层实现            │  ← 复杂逻辑隐藏在内
│                     │
└─────────────────────┘
```

**浅模块** = 大接口 + 少量实现（避免）：

```
┌─────────────────────────────────┐
│         大型接口                 │  ← 方法多、参数复杂
├─────────────────────────────────┤
│  薄薄的实现                      │  ← 只是透传
└─────────────────────────────────┘
```

设计接口时问自己：

- 我能减少方法数量吗？
- 我能简化参数吗？
- 我能将更多复杂性藏在内部吗？

## 原则

- **深度是接口的属性，不是实现的属性。** 一个深模块内部可以由小的、可 mock 的、可替换的部件组成——它们只是不属于接口。一个模块可以有**内部 seam**（对其实现私有，用于自身测试）以及其接口处的**外部 seam**。
- **删除测试。** 想象删除这个模块。如果复杂性消失了，它只是一个透传。如果复杂性重新出现在 N 个调用者中，它在挣自己的存在价值。
- **接口就是测试表面。** 调用者和测试跨越同一个 seam。如果你想_绕开_接口测试，模块的形状可能有问题。
- **一个适配器 = 假设性的 seam。两个适配器 = 真实的 seam。** 除非有东西确实跨越 seam 变化，否则不要引入 seam。

## 面向可测试性设计

好的接口让测试变得自然：

1. **接受依赖，而不是创建它们。**

    ```typescript
    // 可测试
    function processOrder(order, paymentGateway) {}

    // 难以测试
    function processOrder(order) {
    	const gateway = new StripeGateway();
    }
    ```

2. **返回结果，而不是产生副作用。**

    ```typescript
    // 可测试
    function calculateDiscount(cart): Discount {}

    // 难以测试
    function applyDiscount(cart): void {
    	cart.total -= discount;
    }
    ```

3. **表面积小。** 方法越少 = 需要的测试越少。参数越少 = 测试设置越简单。

## 关系

- 一个**模块**只有一个**接口**（它呈现给调用者和测试的表面）。
- **深度**是**模块**相对于其**接口**的属性。
- **Seam** 是**模块**的**接口**所在之处。
- **适配器**位于**Seam**处，满足**接口**。
- **深度**为调用者产生**杠杆效应**，为维护者产生**局部性**。

## 已拒绝的框架

- **深度作为实现行数与接口行数的比率**（Ousterhout）：会奖励膨胀实现。我们使用深度即杠杆效应。
- **"接口"作为 TypeScript 的 `interface` 关键字或类的公有方法**：过于狭窄——这里的接口包括调用者必须知道的每个事实。
- **"边界（Boundary）"**：与 DDD 的 bounded context 过载。用 **seam** 或 **interface**。
