一个技能的存在是为了从随机系统中提取确定性。**可预测性（Predictability）**——agent 每次运行采用相同的_过程_，而不是产生相同的输出——是根本美德；下面的每个杠杆都服务于它。

**粗体术语**在 [`GLOSSARY.md`](GLOSSARY.md) 中定义；在那里查找完整含义。

## 调用方式

两种选择，权衡不同的成本：

- **Model-invoked** 技能保留 **description**，这样 agent 可以自主触发它，_并且_其他技能可以调用它（你仍然可以输入它的名称）。它贡献 **context load**——description 每回合都在窗口中。机制：省略 `disable-model-invocation`，并编写面向模型的 description，带有丰富的触发短语（"当用户想要...、提到..."时使用）。
- **User-invoked** 技能从 agent 的可达范围内移除 description：只有你输入它的名称才能调用它——其他技能不能。零 context load，但花费 **cognitive load**：_你_是必须记住它存在的索引。机制：设置 `disable-model-invocation: true`；`description` 变成面向人类的——一行摘要，触发列表被剥离。

仅当 agent 必须自己触发该技能时，或另一个技能必须调用它时，才选择 model-invocation。如果它仅由手动触发，使其 user-invoked 并支付零 context load。

当 user-invoked 技能多到你记不过来时，累积的 cognitive load 由一个 **router skill** 解决：一个 user-invoked 技能命名其他技能以及何时使用每个。

## 编写 description

Model-invoked 的 **description** 做两件事——说明技能是什么，并列出应触发它的**分支**。每个词增加 **context load**，因此 description 需要比正文更严格的修剪：

- **将技能的领先词放在最前面**——description 是它执行调用工作的地方。
- **每个分支一个触发器。** 重命名单个分支的同义词是**重复**——"使用 TDD 构建功能……要求测试优先开发"是用两种方式写的一个分支。合并；只保留真正不同的分支。
- **删除正文中已有的标识。** 将 description 限制为触发器，加上任何"当其他技能需要..."的调用条款。

## 信息层次

一个技能由两种内容类型构建——**步骤**和**参考**——它们可以自由混合：一个技能可以全是步骤、全是参考、或两者兼有。核心决策是使用哪个，以及每个在**信息层次**中的位置——一个按 agent 需要材料的紧迫性排序的梯子：

1. **技能内步骤**——`SKILL.md` 中的有序操作，主要层级：agent 做什么，按顺序。每个步骤以**完成标准**结束，告诉 agent 工作完成的条件。使其_可检查_（agent 能否区分已完成与未完成？）并在重要时_详尽_（"每个修改过的模型都已说明"，而不是"生成变更列表"）——模糊的标准会招致**过早完成**。
2. **技能内参考**——`SKILL.md` 中的定义、规则或事实，按需查阅。通常是一个合法的扁平同级集（审查的每条规则在同一个梯级上）——很好的安排，不是坏味道。
3. **外部参考**——从 `SKILL.md` 推送到单独文件的参考，通过**上下文指针**访问，仅在指针触发时加载。（涵盖_已披露的_参考——兄弟文件如 `GLOSSARY.md`，仍然是技能的一部分——到完全**外部参考**，存在于技能系统之外，任何技能都可以指向。）

要求高的完成标准驱动彻底的**legwork**——agent 在工作中进行的挖掘——无论技能是否有步骤，因为"每条规则都应用了"约束了扁平参考，就像"每个步骤都完成了"约束了序列。

推送太少，顶部臃肿；推送太多，你隐藏了 agent 实际需要的材料。这种张力是整个决策。

**渐进式披露（Progressive disclosure）** 是向下移动梯子的动作——从 `SKILL.md` 出来进入链接文件——使顶部保持清晰。机制：技能文件夹中的一个链接 `.md` 文件，以其内容命名。有些技能不只以一种方式使用，每种不同的方式是一个**分支**——不同的运行采用不同的技能路径。分支是最清晰的披露测试：内联每个分支都需要的内容，将只有某些分支触及的内容推到指针后面。**上下文指针**的_措辞_，而不是其目标，决定 agent 何时以及多可靠地触及材料。

梯子决定_内容放多深_，而**共置（Co-location）**决定_一旦放好，什么放在它旁边_：将一个概念的定义、规则和注意事项保持在一个标题下而不是分散，以便阅读一个部分时带来其邻近部分。

## 何时拆分

**粒度**是你划分技能的精细程度，每个切分花费两种负载之一，因此仅当切分值得时才拆分。两种切分：

- **按调用方式**——当你有一个应独立触发的不同**领先词**，或另一个技能必须调用它时，拆分出 **model-invoked** 技能。你支付**context load**给新的始终加载的**description**，因此独立的可达性必须值得。
- **按顺序**——当后续步骤（一个步骤的**后完成步骤**）诱使 agent 仓促对待当前步骤（**过早完成**）时，拆分一系列**步骤**。将它们保持在视野之外鼓励 agent 在当前任务上做更多**legwork**。

## 修剪

将每个含义保持在**单一真相来源**：一个权威位置，这样更改行为是一次性编辑。

检查每一行的**相关性**：它是否仍然与技能的功能有关？

然后逐句搜索**无操作（no-op）**，而不仅仅是逐行：对每个句子单独运行无操作测试，当失败时，删除整个句子而不是从中修剪词语。要积极——大多数失败的文章应该被删除，而不是重写。

## 领先词

一个**领先词（leading word）**是一个已经存在于模型预训练中的紧凑概念，agent 在运行技能时用它来思考（例如_lesson_、_fog of war_、_tracer bullets_）。在文本中重复出现（尽管不一定——一个强的领先词可能只需出现一次），它累积了一个分布式定义，并通过招募模型已有的先验知识，在最少的 token 中锚定整个行为区域。

它为可预测性服务两次。在正文中它锚定_执行_：每次出现该词时，agent 达到相同的行为。在 description 中它锚定_调用_：当相同的词存在于你的提示、文档和代码中时，agent 将共享语言链接到技能并更可靠地触发它。

寻找将技能重构为使用领先词的机会。在三个位置拼出的三元组（**重复**），花一个句子指向一个想法的 description——每个都是一个等待**折叠**成单个 token 的段落。示例包括：

- "fast, deterministic, low-overhead" -> _tight_ — 一个跨阶段重新陈述的品质——折叠成一个预训练的词（一个_tight_回路）。
- "a loop you believe in" -> _red_ — 将模糊的门转换为二进制的可观察状态（回路在 bug 上变_red_，或者不变）。

你赢两次：更少的 token，_和_更尖锐的钩子供 agent 挂载其思考。假设每个技能都携带了领先词可以退休的重述——去找它们。

## 失败模式

用这些来诊断用户可能遇到的技能问题。

- **过早完成（Premature completion）**——在真正完成之前结束一个步骤，注意力滑到_已完成_。防御，按顺序：首先锐化完成标准（便宜、本地）；只有当它不可减少地模糊_并且_你观察到仓促时，才通过拆分隐藏后完成步骤（顺序切分）。
- **重复（Duplication）**——同一含义在多个位置。花费维护和 token，并将含义在梯子上的突出度推过其实际等级。
- **沉积（Sediment）**——沉积在老层中，因为添加感觉安全而移除感觉有风险。任何没有修剪纪律的技能的默认命运。
- **蔓延（Sprawl）**——一个技能简单地说太长，即使每一行都是活跃和独特的。伤害可读性和可维护性，并浪费 token。治疗方法是梯子：将**参考**披露到指针后面，并按**分支**或顺序拆分，使每条路径只携带它需要的。
- **无操作（No-op）**——一行 agent 默认已经遵守的指令，所以你支付负载来说没什么。测试：它是否改变了与默认相比的行为？一个弱的领先词（当 agent 已经相当彻底时还说_be thorough_）是一个无操作；修复是一个更强的词（_relentless_），而不是不同的技术。
- **否定（Negation）**——通过禁止来引导适得其反：_不要想大象_命名了大象并使其更可用，而不是更少。提示**正面**——陈述目标行为，使被禁止的从未被提及；仅在无法正面表述的硬性护栏上保留禁止，即使如此也要与做什么配对。
