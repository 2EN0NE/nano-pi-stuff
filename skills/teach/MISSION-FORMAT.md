# MISSION.md 格式

`MISSION.md` 位于工作区根目录。它捕获用户学习此主题的_原因_。每项教学决策——下一步教什么、提供哪些资源、设计哪些练习——都应追溯到此文档。

## 模板

```md
# 任务：{主题}

## 为什么

{1-3 句话。用户追逐的具体现实世界目标。当他们掌握此技能后，生活或工作会有哪些改变？避免 "为了理解 X" 之类的抽象表述——追问底层成果。}

## 成功的标志

- {一个具体的、可观察的用户将能做的事情}
- {另一个具体的事情}
- {……}

## 约束

- {时间、预算、先前承诺、学习偏好、任何影响方法边界的因素}

## 范围外

- {用户明确当前不想追逐的邻近主题——保护最近发展区}
```

## 规则

- **每个工作区一个任务。** 如果用户想学两件不相关的事情，那应该是两个工作区。
- **具体优于抽象。** "十月前跑完半马"胜过"变得健康"。"向团队交付一个 Rust CLI 工具"胜过"学 Rust"。
- **对模糊性保持警惕。** 如果用户无法阐明为什么，在写任何内容之前先追问他们。糟糕的任务比没有任务更坏。
- **当现实变化时修订。** 任务会变化。当用户的目标移动时，更新此文件——不要让过时的任务引导未来的学习。
- **保持简短。** 如果 `MISSION.md` 超过一屏，它就不再是指南针，而是计划了。
