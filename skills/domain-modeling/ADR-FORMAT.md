# ADR 格式

ADR 存放在 `docs/adr/` 中，使用顺序编号：`0001-slug.md`、`0002-slug.md` 等。

懒创建 `docs/adr/` 目录——只有在需要第一个 ADR 时才创建。

## 模板

```md
# {决策的简短标题}

{1-3 句话：上下文是什么，我们决定了什么，以及为什么。}
```

就这样。一个 ADR 可以是单个段落。价值在于记录*做出了*一个决策以及*为什么*——而非填写章节。

## 可选部分

仅当它们增加真正价值时才包含。大多数 ADR 不需要。

- **状态**（Status）前言（`proposed | accepted | deprecated | superseded by ADR-NNNN`）——当决策被重新审视时有用
- **考虑过的选项**（Considered Options）——仅当被拒绝的替代方案值得记住时
- **后果**（Consequences）——仅当需要指出不明显的下游影响时

## 编号

扫描 `docs/adr/` 以找到最高现有编号并加一。

## 何时提供 ADR

以下三个条件必须全部为真：

1. **难逆转**——以后改变主意的成本有意义
2. **无上下文时令人惊讶**——未来的读者会看着代码好奇"他们到底为什么这么做？"
3. **真实权衡的结果**——存在真正的替代方案，你因特定原因选择了一个

如果一个决策容易逆转，跳过它——你只会逆转它。如果它不令人惊讶，没人会好奇为什么。如果没有真正的替代方案，除了"我们做了显然的事"，没什么可记录的。

### 什么符合条件

- **架构形状。** "我们在用 monorepo。""写模型是事件源的，读模型被投射到 Postgres。"
- **上下文间的集成模式。** "Ordering 和 Billing 通过领域事件通信，而不是同步 HTTP。"
- **带来锁定成本的技术选择。** 数据库、消息总线、认证提供者、部署目标。不是每个库——只是那些换掉需要一个季度的。
- **边界和范围决策。** "客户数据由 Customer 上下文拥有；其他上下文仅通过 ID 引用。" 明确的否定与肯定同样有价值。
- **对显然路径的故意偏离。** "我们使用手动 SQL 而不是 ORM，因为 X。" 任何合理的读者可能假设相反的情况。这阻止下一个工程师"修复"某个故意为之的事情。
- **代码中不可见的约束。** "由于合规要求，我们不能使用 AWS。""由于合作伙伴 API 合同，响应时间必须在 200ms 以下。"
- **被拒绝的替代方案——当拒绝的原因不明显时。** 如果你考虑过 GraphQL 因为微妙的原因选择了 REST，记录下来——否则六个月后又会有人建议 GraphQL。
