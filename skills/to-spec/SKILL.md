本技能获取当前对话上下文和代码库理解，并生成一个 spec（你可能知道这个文档叫 PRD）。**不要**采访用户——只需综合你已经知道的内容。

issue tracker 和 triage 标签词汇应已提供给你——如果未提供，运行 `/setup-matt-pocock-skills`。

## 流程

1. 探索代码库以了解代码的当前状态（如果你还没有）。在整个 spec 中使用项目领域词汇表的语言，并尊重所涉及区域中的 ADR。

2. 勾画出你将要测试功能的 seam。现有 seam 优先于新 seam。尽可能使用最高的 seam。如果需要新 seam，在你能达到的最高点提出。整个代码库的 seam 越少越好——理想数量是一个。

    与用户确认这些 seam 是否符合他们的期望。

3. 使用下面的模板编写 spec，然后发布到项目 issue tracker。应用 `ready-for-agent` triage 标签——无需额外的分类。

<spec-template>

## 问题陈述

用户面临的问题，从用户的角度出发。

## 解决方案

问题的解决方案，从用户的角度出发。

## 用户故事

一个长的、编号的用户故事列表。每个用户故事的格式应为：

1. 作为一个<角色>，我想要一个<功能>，以便于<收益>

<user-story-example>
1. 作为一个手机银行客户，我想查看我的账户余额，以便于做出更明智的消费决策
</user-story-example>

用户故事列表应非常详尽，涵盖功能的所有方面。

## 实现决策

做出的实现决策列表。可以包括：

- 将要构建/修改的模块
- 将要修改的模块接口
- 来自开发者的技术澄清
- 架构决策
- 模式变更
- API 契约
- 特定交互

**不要**包含具体的文件路径或代码片段。它们可能很快过时。

例外：如果原型产生了比散文更精确地编码决策的片段（状态机、reducer、模式、类型形状），将其内联在相关决策中并简要说明它来自原型。裁剪到决策丰富的部分——不是工作演示，只是重要的部分。

## 测试决策

做出的测试决策列表。包括：

- 好测试的描述（只测试外部行为，不测试实现细节）
- 将要测试的模块
- 测试的参考先例（即代码库中类似类型的测试）

## 范围外

此 spec 的范围外的事物描述。

## 进一步说明

关于该功能的任何其他说明。

</spec-template>
