# 原型

原型是**回答问题的可丢弃代码**。问题决定形状。

## 选择分支

确定正在回答哪个问题——从用户的提示、周围代码或询问用户（如果用户在场）：

- **"这个逻辑/状态模型对吗？"** → [LOGIC.md](LOGIC.md)。构建一个微小的交互式终端应用，通过难以在纸上推理的案例来推动状态机。
- **"这个应该看起来像什么？"** → [UI.md](UI.md)。在单个路由上生成几种根本不同的 UI 变体，通过 URL 搜索参数和浮动底栏切换。

两个分支产生非常不同的产物——搞错分支浪费整个原型。如果问题确实模糊且用户不可达，默认选择与周围代码更匹配的分支（后端模块 → 逻辑；页面或组件 → UI）并在原型顶部说明假设。

## 两者都适用的规则

1. **从第一天起就是一次性的，并明确标记。** 将原型代码放在实际使用的位置附近（在它正在为其做原型的模块或页面旁边），使上下文显而易见——但命名方式能让随便的读者看出它是原型而不是生产代码。对于一次性 UI 路由，遵守项目已使用的任何路由约定；不要发明新的顶层结构。
2. **一条命令即可运行。** 项目的现有任务运行器支持的任何方式——`pnpm <name>`、`python <path>`、`bun <path>` 等。用户必须能不假思索地启动它。
3. **默认不持久化。** 状态存在于内存中。持久化是原型正在_检查_的东西，而不是它应依赖的东西。如果问题明确涉及数据库，使用一个临时 DB 或带有明确"PROTOTYPE — wipe me"名称的本地文件。
4. **跳过打磨。** 没有测试，没有超出使原型_可运行_的错误处理，没有抽象。重点是快速学到东西。
5. **展现状态。** 在每次操作（逻辑）后或每次变体切换（UI）时，打印或渲染完整的相关状态，以便用户看到什么发生了变化。
6. **完成后捕获。** 将任何验证过的决策折叠到真实代码中，然后将原型本身作为**一级来源**捕获：提交到一个一次性分支，不在 main 上，并在实现 issue 上留下指向该分支的上下文指针。同时捕获答案——结论和它解决的问题——在 issue 或 commit 中。主分支只保留验证过的决策。
