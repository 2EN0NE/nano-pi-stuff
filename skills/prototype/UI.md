# UI 原型

在单个路由上生成**几种根本不同的 UI 变体**，通过浮动底栏切换。用户在浏览器中翻看变体，选择一个（或从每个中偷取部分），然后丢弃其余。

如果问题是关于逻辑/状态而不是某物的外观——错误的分支。使用 [LOGIC.md](LOGIC.md)。

## 何时选择这个分支

- "这个页面应该看起来像什么？"
- "我想在决定之前看几个这个仪表板的选项。"
- "为设置页面尝试不同的布局。"
- 当用户本会花一整天在脑中在三个模糊的线框图之间做选择时。

## 两个子形状——强烈偏好子形状 A

当UI原型**与应用的其余部分放在一起**时更容易判断——真实的头部、真实的侧边栏、真实的数据、真实的密度。一个孤立的一次性路由是一个真空：每个变体单独看都很好。只要存在一个合理的现有页面来托管变体，就默认用子形状 A。只有当原型确实没有附近的家时才选择子形状 B。

### 子形状 A — 调整现有页面（首选）

路由已经存在。变体在**同一路由**上渲染，通过 `?variant=` URL 搜索参数门控。现有的数据获取、参数和认证都保留——只有渲染部分切换。这是默认选项；除非有特定原因不这样做，否则选这个。

如果原型针对的还没有页面但*自然应存在于一个页面内部*（仪表板的新部分、设置屏幕上的新卡片、现有流程中的新步骤）——仍然是子形状 A。在宿主页面内部挂载变体。

### 子形状 B — 新页面（最后手段）

只有在被原型化的东西确实没有现有页面可以放置在内部时才使用这个——例如一个全新的顶层界面，或一个无法在任何合理位置嵌入的流程。

创建一个**一次性路由**，遵循项目已有的任何路由约定——不要发明新的顶层结构。命名方式使它显然是原型（例如在路径或文件名中包含 `prototype` 字样）。相同的 `?variant=` 模式。

在提交到子形状 B 之前，做合理性检查：真的没有现有页面可以嵌入吗？空路由隐藏了填充页面才会暴露的设计问题。

在两个子形状中，浮动底栏是相同的。

## 流程

### 1. 陈述问题并选择 N

默认**3 个变体**。超过 5 个就不再是根本不同了，开始变成噪音——限制在此。

用一行写计划，放在原型位置或文件顶部注释中：

> "设置页面的三个变体，通过 `?variant=` 切换，在现有的 `/settings` 路由上。"

无论用户是否在场提出反对意见，这都有效。

### 2. 生成根本不同的变体

起草每个变体。坚持每个：

- 页面的目的及其可访问的数据。
- 项目的组件库/样式系统（TailwindCSS、shadcn、MUI、普通 CSS 等）。
- 清晰的导出组件名称，如 `VariantA`、`VariantB`、`VariantC`。

变体必须是**结构上不同**的——不同的布局、不同的信息层次、不同的主要操作，而不仅仅是不同的颜色。三个略微调整的卡片网格不是 UI 原型，而是壁纸。如果两个草稿太相似，用明确的"不要使用卡片网格"指导重做一个。

### 3. 将它们连接在一起

在路由上创建一个单一的切换组件：

```tsx
// 伪代码——适应项目的框架
const variant = searchParams.get('variant') ?? 'A';
return (
	<>
		{variant === 'A' && <VariantA {...data} />}
		{variant === 'B' && <VariantB {...data} />}
		{variant === 'C' && <VariantC {...data} />}
		<PrototypeSwitcher variants={['A', 'B', 'C']} current={variant} />
	</>
);
```

对于子形状 A（现有页面）：保持切换器之上的所有现有数据获取；只有渲染的子树按变体变化。

对于子形状 B（新页面）：`/prototype/<name>` 下的一次性路由挂载相同的切换器。

### 4. 构建浮动切换器

一个小的固定位置栏，在屏幕底部居中，有三个部分：

- **左箭头** —— 循环到前一个变体（环绕）。
- **变体标签** —— 显示当前变体键，如果变体导出了名称，也显示名称。如 `B — Sidebar layout`。
- **右箭头** —— 向前循环（环绕）。

行为：

- 点击箭头更新 URL 搜索参数（使用框架的路由器——Next 的 `router.replace`、React Router 的 `navigate` 等），使变体可分享且刷新稳定。
- 键盘：`←` 和 `→` 箭头键也循环。当 `<input>`、`<textarea>` 或 `[contenteditable]` 聚焦时不要拦截箭头键。
- 视觉上与页面区分（如高对比度药丸、微妙的阴影），明显不属于正在评估的设计的一部分。
- 在生产构建中隐藏——通过 `process.env.NODE_ENV !== 'production'` 或等价检查门控，这样不小心的原型合并不会将栏发布给用户。

将切换器放在一个共享组件中，使两个子形状都能重用它。放在项目中共享 UI 所在的任何位置。

### 5. 交给用户

给出 URL（和 `?variant=` 键）。用户会在有空时翻看。有趣的反馈通常是**"我想要 B 的头部和 C 的侧边栏"**——这才是他们真正想要的设计。

### 6. 捕获答案并清理

一旦一个变体获胜，捕获答案——哪个变体和为什么——然后按 [SKILL](SKILL.md) 描述的方式捕获原型。将胜者折叠到真实代码中，将其余部分移到一次性分支上，而不是 main 上：

- **子形状 A** —— 将胜者折叠到现有页面中；从 main 中删除失败的变体和切换器。
- **子形状 B** —— 将获胜变体提升为真实路由；从 main 中删除一次性路由和切换器。

完整变体集是一级来源，因此它落在一性次分支上，而不是垃圾箱——变体组件和切换器留在主分支中会迅速腐烂并混淆下一个读者。

## 反模式

- **仅颜色或文案不同的变体。** 那是调整，不是原型。真正的变体在结构上有分歧。
- **变体之间共享太多代码。** 共享 `<Header>` 是可以的；共享 `<Layout>` 就违背了目的。每个变体应自由地丢弃布局。
- **将变体连接到真实变更操作。** 只读原型就可以。如果变体需要变更，指向一个桩——问题是"这应该看起来像什么"，而不是"后端是否工作"。
- **直接将原型提升到生产。** 变体代码是在原型约束下编写的（无测试、最小错误处理）。折叠时正确重写它。
