# 逻辑原型

一个微小的交互式终端应用，让用户手动驱动状态模型。当问题是关于**业务逻辑、状态转换或数据形状**时使用——那种在纸面上看起来合理，但只有在通过真实案例推动时才会感觉不对的东西。

## 何时选择这个分支

- "我不确定这个状态机是否能处理 X 然后 Y 的边界情况。"
- "这个数据模型是否真的允许我表示...的情况"
- "我想在写之前感受一下 API 应该是什么样子"
- 任何用户想**按按钮并观察状态变化**的情况

如果问题是"这应该看起来像什么"——错误的分支。使用 [UI.md](UI.md)。

## 流程

### 1. 陈述问题

在写代码之前，写下你在为哪个状态模型和哪个问题做原型。一段话，放在原型的 README 或文件顶部注释中。回答了错误问题的逻辑原型是纯粹的浪费——使问题明确，以便以后可以检查，无论用户现在在观看还是以后非同步返回。

### 2. 选择语言

使用宿主项目使用的任何语言。如果项目没有明显的运行时（如文档仓库），询问。

匹配项目现有的工具约定——不要仅为原型添加新的包管理器或运行时。

### 3. 将逻辑隔离在可移植模块中

将实际逻辑——正在回答问题的那部分——放在一个小巧、纯的接口后面，以后可以移植到真实的代码库中。围绕它的 TUI 是一次性的；逻辑模块不应如此。

正确的形状取决于问题：

- **纯 reducer** — `(state, action) => state`。当动作是离散事件且状态是单个值时使用。
- **状态机**——显式的状态和转换。当"现在哪些操作是合法的"本身就是问题的一部分时使用。
- **一组纯函数**作用于普通数据类型。当没有隐含的当前状态——只有转换时使用。
- **具有清晰方法表面的类或模块**——当逻辑确实拥有持续的内部状态时使用。

选择最符合所问问题的形状，*而不是*最容易连接到 TUI 的形状。保持纯：无 I/O、无终端代码、无用于控制流的 `console.log`。TUI 导入它并调用它；没有东西流向另一个方向。

这就是使原型在其自身生命周期之外仍然有用的原因：当问题被回答后，验证过的 reducer/机器/函数集可以单独移植到真实模块中。

### 4. 构建暴露状态的最小 TUI

构建为**轻量级 TUI**——每个 tick，清屏（`console.clear()` / `print("\033[2J\033[H")` / 等价）并重新渲染整个帧。用户应始终看到一个稳定的视图，而不是不断增长的滚动缓冲。

每帧有两个部分，按此顺序：

1. **当前状态**，美化打印且 diff 友好（每行一个字段，或格式化的 JSON）。使用**粗体**表示字段名或章节标题，使用**暗淡**表示不太重要的上下文（时间戳、ID、派生值）。原生 ANSI 转义码即可——`\x1b[1m` 粗体、`\x1b[2m` 暗淡、`\x1b[0m` 重置。除非项目中已有样式库，否则不需要引入。
2. **键盘快捷键**，列在底部：`[a] add user  [d] delete user  [t] tick clock  [q] quit`。键用粗体，描述用暗淡，或反之——只要清晰即可。

行为：

1. **初始化状态**——一个单一的内存对象/结构。启动时渲染第一帧。
2. **一次读取一个按键（或一行）**，分发到处理程序以变更状态。
3. **重新渲染**每次操作后的完整帧——不要追加，替换。
4. **循环直到退出。**

整个帧应适应一个屏幕。

### 5. 使其一条命令可运行

将脚本添加到项目现有的任务运行器（`package.json` scripts、`Makefile`、`justfile`、`pyproject.toml`）。用户应运行 `pnpm run <prototype-name>` 或等价命令——永远不需要记住路径。

如果宿主项目没有任务运行器，只需将命令放在原型 README 的顶部。

### 6. 交给用户

给用户运行命令。他们自己驱动；有趣的时刻是他们说"等等，那应该是不可能的"或"哈，我假设 X 会是不同的"——这些是*想法*中的 bug，而这就是全部意义所在。如果他们想要添加新操作，添加。原型会演化。

### 7. 捕获答案和原型

一旦原型回答了问题，捕获答案，然后按 [SKILL](SKILL.md) 描述的方式捕获原型。逻辑特定映射：验证过的 reducer/机器/函数集移植到真实模块（决策，被吸收）；TUI 外壳随一次性分支一起保存，该分支保留原型作为一级来源。

## 反模式

- **不要添加测试。** 需要测试的原型不再是原型。
- **不要连接到真实数据库。** 使用内存存储，除非问题特别关于持久化。
- **不要泛化。** 没有"如果我们以后想支持 X 的话。"原型回答一个问题。
- **不要将逻辑和 TUI 混在一起。** 如果 reducer/状态机引用 `console.log`、提示或终端转义码，它就不再可移植。将 TUI 保持为纯模块上的薄壳。
- **不要将 TUI 外壳发布到生产。** 外壳是为终端手动驱动而优化的。背后的逻辑模块才是值得保留的部分。
