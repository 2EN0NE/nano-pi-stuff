# 诊断 Bug

一套针对疑难 bug 的纪律。只有在明确有理由的情况下才跳过阶段。

在探索代码库时，阅读 `CONTEXT.md`（如果存在）以获得相关模块的清晰心智模型，并检查所涉及区域的 ADR。

## 阶段 1 — 构建反馈回路

**这是本技能的核心。** 其他一切都是机械的。如果你有一个针对该 bug 的**紧致**通过/失败信号——一个能在_这个_ bug 上变红的信号——你就能找到原因；二分查找、假设检验、插桩都只是消耗它。如果你没有这样的信号，再多的代码阅读也救不了你。

在这里投入不成比例的努力。**要激进。要有创造力。拒绝放弃。**

### 构建反馈回路的方法——大致按此顺序尝试

1. **失败测试**——在触及 bug 的任何 seam 处——单元、集成、e2e。
2. **Curl / HTTP 脚本**——针对运行中的开发服务器。
3. **CLI 调用**——使用固定的输入，将 stdout 与已知良好的快照进行 diff。
4. **无头浏览器脚本**（Playwright / Puppeteer）——驱动 UI，在 DOM/控制台/网络上断言。
5. **重放捕获的追踪。** 将真实的网络请求/载荷/事件日志保存到磁盘；在隔离中通过代码路径重放它。
6. **一次性测试工具。** 启动系统的最小子集（一个服务，mock 的依赖项），通过一个函数调用就能执行 bug 代码路径。
7. **属性/模糊循环。** 如果 bug 是"有时输出错误"，运行 1000 个随机输入并查找失败模式。
8. **二分测试工具。** 如果 bug 在两个已知状态（commit、数据集、版本）之间出现，自动化"在状态 X 启动，检查，重复"，以便可以 `git bisect run`。
9. **差分循环。** 通过旧版本 vs 新版本（或两个配置）运行相同输入并 diff 输出。
10. **人在环中 bash 脚本。** 最后手段。如果人必须点击，用 `scripts/hitl-loop.template.sh` 驱动_他们_。捕获的输出反馈给你。

构建正确的反馈回路，bug 就 90% 修好了。

### 收紧回路

把回路当作产品对待。一旦你有了_一个_回路，**收紧**它：

- 我能让它更快吗？（缓存设置、跳过不相关的初始化、缩小测试范围。）
- 我能让信号更清晰吗？（断言特定症状，而非"没有崩溃"。）
- 我能让它更确定吗？（固定时间、播种 RNG、隔离文件系统、冻结网络。）

一个 30 秒的不稳定回路几乎不比没有回路好；一个 2 秒的确定回路就是紧致的——一种调试超能力。

### 非确定性 bug

目标不是完美的复现而是**更高的复现率**。循环触发 100 次、并行化、增加压力、缩小时间窗口、注入 sleep。50% 抖动的 bug 是可调试的；1% 则不是——不断提高复现率直到可调试。

### 当你确实无法构建回路时

停下来并明确说。列出你尝试过的。向用户请求：(a) 访问复现问题的任何环境，(b) 捕获的产物（HAR 文件、日志转储、核心转储、带时间戳的屏幕录制），或 (c) 添加临时生产插桩的权限。**不要**在没有回路的情况下继续提出假设。

### 完成标准——一个变红的紧致回路

阶段 1 完成的条件是：回路**紧致**且**能变红**——你能命名**一条命令**（脚本路径、测试调用、curl），你**已经至少运行过一次**（粘贴调用及其输出），并且：

- [ ] **能变红**——它驱动实际的 bug 代码路径并断言**用户的确切症状**，所以它能在该 bug 上变红，在修复后变绿。不仅仅是"不报错运行"——它必须能_捕捉这个特定的 bug_。
- [ ] **确定**——每次运行相同判定（不稳定的 bug：固定的高复现率，如上所述）。
- [ ] **快速**——几秒，不是几分钟。
- [ ] **Agent 可运行**——你可以无人值守运行；人在环中仅在 `scripts/hitl-loop.template.sh` 中。

如果你发现自己在回路存在之前就开始阅读代码构建理论，**停下来——直接跳到假设正是本技能要预防的失败。** 没有能变红的命令，就没有阶段 2。

## 阶段 2 — 复现 + 最小化

运行回路。观察它变红——bug 出现。

确认：

- [ ] 回路产生了**用户**描述的失败模式——不是一个碰巧在附近的不同的失败。错误的 bug = 错误的修复。
- [ ] 失败在多次运行中可复现（或非确定性 bug 有足够高的复现率以进行调试）。
- [ ] 你已经捕获了确切症状（错误消息、错误输出、慢速时间），以便后续阶段可以验证修复是否实际解决了问题。

### 最小化

一旦它变红，将复现缩小到**仍然变红的最小场景**。**一次一个**地削减输入、调用者、配置、数据和步骤，每次削减后重新运行回路——只保留对失败关键的部分。

为什么这么麻烦：最小复现缩小了阶段 3 的假设空间（剩余的可疑变量更少），并成为阶段 5 的清晰回归测试。

当**每个剩余元素都是关键性的**时完成——删除其中任何一个都会使回路变绿。

在已复现**且**已最小化之前，不要继续。

## 阶段 3 — 提出假设

在测试任何一个之前，生成 **3–5 个排序后的假设**。单假设生成会锚定在第一个合理的想法上。

每个假设必须是**可证伪的**：陈述它所做的预测。

> 格式："如果 <X> 是原因，那么 <改变 Y> 将使 bug 消失 / <改变 Z> 将使 bug 更严重。"

如果你无法陈述预测，该假设只是一个感觉——丢弃或精炼它。

**在测试之前向用户展示排序后的列表。** 他们通常具有能立即重新排序的领域知识（"我们刚刚部署了对 #3 的变更"），或知道他们已经排除的假设。便宜的检查点，节省大量时间。如果用户不在，则按你的排序继续。

## 阶段 4 — 插桩

每个探针必须映射到阶段 3 中的一个具体预测。**一次只改变一个变量。**

工具偏好：

1. **调试器 / REPL 检查**——如果环境支持。一个断点胜过十行日志。
2. **定向日志**在区分假设的边界处。
3. 绝不"记录所有东西然后 grep"。

**给每个调试日志加上唯一前缀标签**，如 `[DEBUG-a4f2]`。最后的清理就变成一个 grep。未标记的日志存活；标记的日志被删除。

**性能分支。** 对于性能回归，日志通常是错误的。应建立基线测量（计时工具、`performance.now()`、profiler、查询计划），然后二分查找。先测量，后修复。

## 阶段 5 — 修复 + 回归测试

在**修复之前**写回归测试——但仅在存在**正确的 seam** 时。

正确的 seam 是测试能按实际调用点发生的方式演练真实的 bug 模式。如果唯一可用的 seam 太浅（当 bug 需要多个调用者时只有一个调用者测试，无法复制触发 bug 的链条的单元测试），那里的回归测试会产生虚假的信心。

**如果不存在正确的 seam，这个发现本身就是结论。** 记录下来。代码库架构阻止了该 bug 被锁定。标记给下一阶段。

如果存在正确的 seam：

1. 将最小化的复现转换为该 seam 处的失败测试。
2. 观察它失败。
3. 应用修复。
4. 观察它通过。
5. 针对原始（未最小化的）场景重新运行阶段 1 的反馈回路。

## 阶段 6 — 清理 + 事后分析

声明完成前必须完成：

- [ ] 原始复现不再复现（重新运行阶段 1 的回路）
- [ ] 回归测试通过（或 seam 缺失已记录）
- [ ] 所有 `[DEBUG-...]` 插桩已移除（`grep` 该前缀）
- [ ] 一次性原型已删除（或移到明确标记的调试位置）
- [ ] 最终被证明正确的假设记录在 commit / PR 消息中——这样下一个调试者可以学习

**然后问：什么可以防止这个 bug？** 如果答案涉及架构变更（没有好的测试 seam、调用者纠缠不清、隐藏的耦合），将具体内容交给 `/improve-codebase-architecture` 技能。在修复完成**之后**而非之前提出建议——你现在拥有的信息比开始时更多。
