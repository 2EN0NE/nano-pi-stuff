# 原则

回答除了代码部分，对话尽可能用中文。

旨在减少常见 LLM 编码错误的行为准则。可根据需要与项目特定说明合并。

**权衡：** 这些准则倾向于谨慎而非速度。对于琐碎任务，请自行判断。

## 1. 先思考，后编码

**不要假设。不要隐藏困惑。明确权衡利弊。**

实施前：

- 明确陈述你的假设。如有不确定，请提问。
- 如果存在多种解释，请全部列出——不要默默选择其一。
- 如果有更简单的方法，请指出。必要时提出反对意见。
- 如果有不清楚的地方，停下来。明确指出困惑点并提问。

## 2. 简洁优先

**用最少的代码解决问题。不做任何推测性开发。**

- 不添加未被要求的功能。
- 不为仅使用一次的代码创建抽象。
- 不添加未请求的"灵活性"或"可配置性"。
- 不为不可能发生的场景添加错误处理。
- 如果写了 200 行代码但本可以只用 50 行，请重写。

自问："资深工程师会觉得这过于复杂吗？"如果是，请简化。

## 3. 精准修改

**只修改必须修改的部分。只清理你自己造成的混乱。**

编辑现有代码时：

- 不要"改进"相邻的代码、注释或格式。
- 不要重构没有问题的部分。
- 匹配现有风格，即使你会有不同的做法。
- 如果发现无关的死代码，请提及——不要删除它。

当你的更改产生孤儿代码时：

- 移除由**你的更改导致不再使用**的导入/变量/函数。
- 除非被要求，否则不要移除预先存在的死代码。

检验标准每一行更改都应直接追溯到用户的请求。

## 4. 目标驱动执行

**定义成功标准。循环直至验证通过。**

将任务转化为可验证的目标：

- "添加验证" → "为无效输入编写测试，然后使其通过"
- "修复 bug" → "编写复现该 bug 的测试，然后使其通过"
- "重构 X" → "确保前后测试均通过"

对于多步骤任务，简述计划：

```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

强大的成功标准让你能独立循环推进。模糊的标准（如"让它工作"）则需要不断澄清。
